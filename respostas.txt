2 -O formato de configuração oficial será o simplificado de configTargets.yaml + configMetrics.yaml, ou o formato antigo de config.yaml também deve continuar suportado?

R: O formato de configuração oficial será o simplificado de configTargets.yaml + configMetrics.yaml

3 - A aplicação Go deve apenas consumir arquivos de configuração já gerados, ou também deve manter a capacidade antiga de descobrir/montar targets automaticamente a partir de roots?

4 - Na validação opcional de inicialização, se um target configurado estiver com id divergente do OEM, a aplicação deve falhar, apenas logar warning, ou corrigir em memória e continuar?

5 - Essa validação deve escrever um arquivo corrigido de configuração, ou nunca deve alterar arquivos de entrada?

R: Essa hierarquia é a esperada. Além disso, deve se verificar por exemplo, se não faltou algum oracle_database que segue o padrão de nomeclatura, se os hosts relacionados estão errados, etc..

7 - Targets adicionados manualmente/pesquisa livre pelo frontend devem ser aceitos mesmo sem correlação completa?

R: Sim, targets avulsos podem estar presentes. Com atenção para Racs e PDBS, caso seja esses dois, uma correção adicionando os targets que companham o cluster(oracle_database,hosts, etc..) devem ser adicionados se presentes na API

9 - A compatibilidade de saída exige manter os nomes antigos exatamente em lowercase? O exportador legado faz metric.name.lower().

11 - Para métricas textuais/logs, devemos manter a lógica de só reenviar quando o valor muda, exceto métricas marcadas como contínuas?

13 - A correção de timestamp dos incidentes subtraindo 3 horas ainda é necessária ou isso era workaround do ambiente antigo?

14 - Quais variáveis de ambiente devem ser contrato público? Ex.: EM_BASE_URL, PROTOCOL, USER, TOKEN, OTEL_EXPORT_URL, USE_TARGET_CONFIG, USE_TARGET_CACHE.

R: as novas configurações de url do target ficam em configTargets.yaml, portanto EM_BASE_URL, PROTOCOL não precisa existir. Alem de disso USE_TARGET_CONFIG, USE_TARGET_CACHE não são necessárias também, pois eram opcões para quando esse proprio aplicativo gerava configuração. Não sei exatamente oque voce quis dizer com "devem ser contrato publico".

16 - A autenticação continua sendo Basic Auth contra o OEM? O TOKEN antigo passa por xisou.check_health; isso deve existir no Go ou podemos simplificar para usuário/senha/token direto?

R: Sobre essa autenticação, esse token é algo gerado a partir da hash do arquivo fonte, onde a função "check_health" reverte um processo de gerar token com hash e senha, e pega a senha a partir do token e hash. Isso é uma solução temporaria que tinha o objetivo de não deixar a senha explicita em variavel de ambiente, substituindo pelo token. Por enquanto essa solução deve ser mantida (no sentido de que essa funcionalidade deve ser mantida). Mas na prática isso acaba sendo um Basic Auth
